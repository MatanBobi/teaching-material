React-Redux · Performance Deep-Dive

Why React-Redux v6 Was Slower Than v5 (and v7 Fixed It)

React-Redux v6 was a bold rewrite built on the brand-new React Context API. On paper it looked cleaner — in practice, it broke the performance trick that made v5 fast, and introduced a render-cascade problem that v7 had to undo.

1. The big idea behind v6

React 16.3 introduced the new Context API. The React-Redux team rewrote v6 to put the entire Redux store inside a React Context value. Every connect()ed component would simply useContext (or contextType) to read state.

The dream: use only React's official primitives, get concurrent-mode safety for free, and remove the legacy subscribe()-and-forceUpdate() machinery.

2. How v5 worked (the "fast" version)

In v5, every connect() wrapper subscribed directly to the Redux store. When an action dispatched, each connected component ran its own mapStateToProps, shallow-compared the result, and only re-rendered if its slice actually changed. Parent components were not involved.

3. How v6 worked (and why it hurt)

In v6, the store was passed through Context.Provider. When the store updated, the Provider's value changed. React Context has one fundamental rule:

Any consumer of a context re-renders whenever the Provider's value changes — there is no way to opt out.

That meant every connected component got woken up on every dispatch, even if its slice of state hadn't changed. The shallow-equality check was still there, but it ran after React had already scheduled the render through the entire tree.

4. The real problem: the "context cascade"

Because Context updates flow top-down through the React tree, v6 had to re-render parents before their children — even if the parent didn't care about the change. Three secondary problems fell out of this:

Problem

1. Wasted work in every parent

In a deep tree, dispatching a single action could wake up dozens of connected ancestors that ultimately bailed out via shallow equality — but only after React had already started reconciling them.

Problem

2. Stale-props bug ("zombie children")

Because parents re-rendered first, a child could receive props derived from old state at the moment it tried to read new state from context — throwing errors when the data it needed had just been deleted.

Problem

3. No { pure: false } escape hatch

v5 let you opt connected components out of the parent re-render requirement. v6 couldn't honor that — Context dictates the order.

Problem

4. Slower than v5 in benchmarks

Mark Erikson's own benchmarks showed v6 was meaningfully slower than v5 on real apps — the opposite of what a rewrite is supposed to deliver.

5. How v7 fixed it

v7 kept the Context idea — but only used it to pass the store reference, not the state itself. The old subscription model came back, hidden behind hooks (useSelector, useDispatch).

v7 — context delivers the store, subscriptions deliver updates

Store dispatch
→
Each useSelector subscriber runs its selector
→
Reference equality check
→
Only changed components re-render

Plus a subscription tree that walks parents → children in the right order, eliminating the stale-props / zombie-child class of bugs.

7. The takeaway for your class

React Context is a notification system, not a state-management system. Any consumer re-renders on every value change — there is no fine-grained subscription. If you put frequently-changing state into Context, every consumer in the tree pays the cost. This is precisely the trap React-Redux v6 fell into, and why libraries like Redux, Zustand, Jotai, and Valtio all bypass Context for the actual state delivery.

8. Were there official benchmarks?

Yes — Mark Erikson (React-Redux maintainer) set up a dedicated reduxjs/react-redux-benchmarks repo to compare versions head-to-head while v6 was being developed. Here are the actual reported numbers.

From the v6 PR final benchmark run (PR #1000)
  • Both the WIP and final v6 builds were slower than v5 in every benchmark scenario.
  • The final v6 build was the fastest of the v6 attempts, but still behind v5.
  • ~20% slower in the worst case: a flat tree of many rapidly-updating connected components.
  • ~2% slower in deeper trees with more typical update patterns.

Erikson's caveat: these are artificial stress-test benchmarks designed to push every version to its limits — they're a relative comparison, not a real-app measurement.

After v6 shipped

Real users reported "noticeable slowdowns in various scenarios" (issue #1164), which is what triggered the v7 roadmap (issue #1177). Dan Abramov and Sebastian Markbage from the React team explicitly told the React-Redux maintainers to go back to direct store subscriptions and use unstable_batchedUpdates() — which is what v7 did.

Sources: Mark Erikson, "Idiomatic Redux: The History and Implementation of React-Redux" (blog.isquaredsoftware.com), PR #1000 benchmark comment, React-Redux GitHub issues #1164 and #1177, and React issue facebook/react#13739 (Context value propagation performance).
