Tap or click any boxed region inside the mock app. Purple frames are Server Components; blue frames are Client Components.

Where it runs

  • Server Components render on the server during the request.
  • Client Components render in the browser (and during SSR for the initial HTML).

Why this matters

  • Server Components ship zero JavaScript to the client.
  • Client Components are needed for interactivity, state, effects, and browser-only APIs.
Tip: A Server Component can import and render a Client Component, but not the other way around — a Client Component can only receive Client Components as children via composition.

S Server Components

Run on the server. They can async/await directly, hit databases, read files, and access secrets. Their output is serialized and streamed to the browser.

They are not interactive — no useState, no useEffect, no event handlers, no browser APIs.

C Client Components

Marked with the "use client" directive at the top of the file. They run in the browser and, during SSR, on the server to produce initial HTML.

Use them whenever you need state, effects, refs, event handlers, or browser-only APIs like window or localStorage.

The Network Boundary

In traditional SSR, the server renders HTML and then ships every component's JavaScript to the browser to hydrate it. The whole tree crosses the network boundary. With React Server Components, the boundary moves inward: only the components marked "use client" need to ship JavaScript. Everything else stays on the server.
