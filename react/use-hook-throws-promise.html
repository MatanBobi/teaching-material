How use(promise) throws to suspend

React's use hook reads a value from a promise. But if the promise hasn't resolved yet, it does something surprising: it throws the promise itself. The nearest <Suspense> catches it, shows a fallback, and re-renders the component once the promise resolves. Press Play to watch the full lifecycle.

Why throw?

React renders are synchronous functions — they can't await. Throwing is the one mechanism that lets a function abort mid-execution and hand control back to whoever called it. React abuses this: use() throws the pending promise so React's render loop can catch it.

Suspense is a try/catch

Conceptually, every <Suspense> boundary wraps its children in a try/catch. If the caught value is a thenable (a promise), Suspense renders the fallback and attaches a .then() to know when to retry.

Re-render, don't resume

When the promise resolves, React doesn't magically resume the half-finished function. It calls the component from scratch. The second time, use() sees a fulfilled promise and returns its value synchronously — no throw this time.

Cache the promise

Because the component re-runs on every render, the promise must be stable across renders (cached, memoized, or passed in as a prop). Creating fetch(...) inline every render starts a new request every time and never resolves consistently.
